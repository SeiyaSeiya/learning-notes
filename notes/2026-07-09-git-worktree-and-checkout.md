# git worktree と checkout の違い

- 学んだきっかけ: `gas-voc-extract`（GASプロジェクト）でClaude Codeにドキュメント作成を依頼した際、作業ディレクトリが `.claude/worktrees/xxx` という見慣れない場所になっていたことから調査。

## 1. 今回の作業環境の実態

- 作業していたディレクトリは、単純なファイルコピーではなく **git worktree**（同じリポジトリの別ブランチを別ディレクトリに展開したもの）だった。
- 本来のディレクトリ（`main`ブランチ）と、そのworktree（`claude/xxx`ブランチ）は、**同じ `.git`（履歴データ）を共有**している。
- worktree側への変更は本来のディレクトリ（`main`）に一切影響しない。取り込むには、ブランチをpush→PR作成→マージ、または `git merge` が必要。
- これはClaude Code自体の標準動作ではなく、セッションを起動している環境（ハーネス）側が、安全のためタスクごとに専用worktree＋ブランチを自動作成する設定になっていたため。素の `claude` コマンドでは、通常は今いるディレクトリに直接書き込む。

## 2. `git branch` の `+` マークの意味

```
+ claude/gas-project-overview-docs-df0ae9
* main
```

- `*` … **今いるディレクトリ**でチェックアウト中のブランチ
- `+` … **別のworktree（別ディレクトリ）**でチェックアウト中のブランチ
- 記号なし … どこにもチェックアウトされていないブランチ
- この記号は、複数worktreeを使っている時だけ意味を持つ表示。

## 3. 「チェックアウト」の意味

- git は内部にコミット（スナップショット）を記録しているが、それ自体は直接見えない・編集できない。
- 「チェックアウト」とは、**指定したブランチのスナップショットの中身を、実際にディスク上のファイルとして展開すること**。
- 1つのディレクトリには1ブランチ分のチェックアウトしか同時にできない（ファイルが競合するため）。

## 4. `git checkout`/`git switch` と `git worktree add` の違い

| コマンド | 動作 |
|---|---|
| `git checkout -b newbranch` / `git switch -c newbranch` | 新ブランチを作成し、**今のディレクトリの中身をそのブランチに切り替える**（元のブランチの内容は消える） |
| `git worktree add <path> -b newbranch` | 新ブランチを作成するが、**今のディレクトリは切り替えない**。別ディレクトリを新設し、そこにだけ新ブランチをチェックアウトする |

- 通常の `checkout`/`switch` は「1つの場所でブランチを入れ替える」動作。
- `worktree add` は「場所を増やして両方同時に存在させる」動作。
- 今回使われたのは後者だったため、本来のディレクトリは終始 `main` のままで、一度も切り替わっていなかった。

## 5. 派生した気づき：セッション間の知識の引き継ぎ

- Claude Codeの自動メモ機能は作業中のプロジェクト（ディレクトリ）に紐づく仕組みのため、別ディレクトリで新しいセッションを開くと自動的には引き継がれない。
- 確実に引き継ぐには、内容を**ファイルとして書き出しておく**のが良い（ファイルはセッションをまたいで残るため）。このリポジトリ（`learning-notes`）自体がその実践。

## 6. GitHubリモートリポジトリの作成とpush

- `gh`（GitHub CLI）はHomebrewで未インストールだったため `brew install gh` で導入。認証は `gh auth login --git-protocol ssh --web` で行い、ブラウザでのワンタイムコード入力が必要だったのは**初回のみ**。以降はターミナルだけでGitHub操作（リポジトリ作成・push等）が完結する。
- `gh repo create --source=. --push` は、このディレクトリが git worktree であることを正しく認識できず失敗した（`current directory is not a git repository` というエラー。gh CLI の既知の制限）。代わりに `gh repo create <name> --public` でリポジトリ作成のみ行い、`git remote add` と `git push` を別途手動で実行する方式に切り替えた。
- リモートURL `git@github.com:user/repo.git` は拡張子付きのファイル名のような見た目だが、実際は**SSH接続用のURL**。`git@` は接続用の固定ユーザー名、`github.com` がホスト、`.git` は慣習的なサフィックス（省略可）。
- `git push origin main` は「今このディレクトリでチェックアウトしているブランチ」ではなく、「指定した名前（`main`）のローカルブランチが指すコミット」をpushする。そのため、別ブランチ（`claude/xxx`）をチェックアウトしているworktreeの中からでも、`main`ブランチをpushできた。
- 今回、`main`と作業用ブランチ（`claude/xxx`）が同じコミットを指していたため、mergeという操作を挟まずにpushだけで済んだ。これは「たまたま両ブランチの内容が一致していたから」であり、`main`が特別扱いされてブランチの内容を自動的に取り込む、という仕組みは存在しない。
- ローカルブランチは、明示的に `git push` しない限りリモートには一切反映されない。「指定しなければmainだけが自動でpushされる」というルールはなく、常に**実行したpushコマンドが指定した対象**だけが反映される。
- ブランチという概念自体はgitの基本機能であり、worktreeとは無関係に存在する。worktreeは「複数のブランチを同時に別ディレクトリへ展開できる」という追加機能であり、「他のブランチの内容を見るために、今の作業を退避してswitchする」という手間を回避する仕組みにすぎない（switchコマンド自体が不要になるわけではない）。
- Claude Codeがタスクごとに専用worktree＋ブランチを自動作成するのは、Claude Code自体の標準動作ではなく、セッションを起動する環境（ハーネス）側の安全設計（本体ブランチを汚さない、レビュー単位を作る、並行タスクの衝突を防ぐ）によるもの。この設計は一般的だが必須ではない。
- 作業用ブランチ（`claude/xxx`）はタスクごとの使い捨てであり、内容が`main`に反映済みであればpushする必要はない。ただし、そのブランチをチェックアウトしているworktreeの中からは、そのブランチ自身を削除できない（`git worktree remove` で先にworktreeを取り除く必要がある）。
