# CLAUDE.mdの役割と活用方法

- 学んだきっかけ: `gas-project`（複数のGASプロジェクトを格納するローカルディレクトリ）にプロジェクト共通ルールを持たせるため、`gas-project/.claude/CLAUDE.md`を作成する過程で、CLAUDE.mdの仕組み全般を調査した。

## 1. CLAUDE.mdとは何か、なぜ精度が上がるか

- CLAUDE.mdはセッション開始時に毎回コンテキストとして自動的に読み込まれる、プロジェクト向けの永続的な指示ファイル。
- ビルド/テストコマンド、命名規則、ディレクトリ構成、「常にこうする」というルールを書いておくことで、Claudeが毎回推測したり、同じ間違いを繰り返したり、同じ指摘を何度も受けたりすることを防げる。
- ただしCLAUDE.mdの内容は「アドバイザリー」（Claudeが解釈して従うかどうかを判断する）であり、強制力はない。絶対に例外なく守らせたい処理（mainへの直push禁止、編集後に必ずlintを走らせる等）はCLAUDE.mdではなく**hook**（決まったタイミングで確実にスクリプトを実行する仕組み）で実装する方が確実。

## 2. 配置場所の選択肢

| スコープ | 場所 | 用途 | 共有範囲 |
|---|---|---|---|
| 組織全体 | 各OSの管理ポリシー配置場所 | IT/DevOpsが管理する全社共通ルール | 組織の全ユーザー |
| ユーザー個人 | `~/.claude/CLAUDE.md` | 全プロジェクト共通の個人的な好み | 自分のみ（全プロジェクト） |
| プロジェクト | `./CLAUDE.md` または `./.claude/CLAUDE.md` | チーム共有のプロジェクトルール | チームメンバー（バージョン管理経由） |
| ローカル個人 | `./CLAUDE.local.md` | このプロジェクトだけの個人設定（サンドボックスURLなど） | 自分のみ（このプロジェクトのみ） |

- `./CLAUDE.md` と `./.claude/CLAUDE.md` は**機能的に完全に同格**（優先度・読み込みタイミングに差はない）。どちらを使うかは整理上の好みの問題。
  - `./CLAUDE.md`: READMEと同じ感覚でルート直下に置き、リポジトリを開いた瞬間に目に入る。
  - `./.claude/CLAUDE.md`: `settings.json`・`.claude/rules/`・`.claude/skills/`など他のClaude関連設定と同じ場所にまとまり、一貫性が出る。既に`.claude/`ディレクトリがあるプロジェクト（例: worktree運用で`.claude/worktrees`が既に存在する等）ではこちらが自然。
- `CLAUDE.local.md`は「チームに共有しない、自分だけの個人設定」用。プロジェクト固有の仕様など、チームメンバーが知っておくべき情報を書く場所ではない（`.gitignore`対象にする）。

## 3. 読み込みの挙動（ディレクトリツリー）

- Claude Codeは起動したディレクトリから**上に**ディレクトリツリーを遡ってCLAUDE.md / CLAUDE.local.mdを探し、見つかった全ファイルを起動時にまとめて読み込む（ルートに近いものから順に、起動ディレクトリに近いものが後に読まれる）。
- 起動ディレクトリの**下**（子ディレクトリ）にあるCLAUDE.mdは起動時には読み込まれず、Claudeがそのサブディレクトリ内のファイルを実際に読みに行ったタイミングで遅延読み込みされる。
- 例：`gas-project/gas-feedback-generate/`で`claude`を起動した場合、`gas-project/.claude/CLAUDE.md`（共通ルール）と`gas-feedback-generate/.claude/CLAUDE.md`（そのプロジェクト固有ルール、もし作れば）の両方が起動時に読まれる。
- `gas-project/`ルートで起動した場合は、ルートのCLAUDE.mdは読まれるが、`gas-feedback-generate/`側のCLAUDE.mdは遅延読み込みになる。

## 4. 効果的な書き方のコツ

- **短さが正義**: 200行を超えると遵守率が落ちる。長すぎるCLAUDE.mdは「大事なルールがノイズに埋もれて無視される」原因になる。
- **判断基準**: 「その一行を消したら、Claudeが間違えるようになるか？」に当てはまらないなら削る。コードを読めば分かること・言語の標準的な慣習・自明な心構えは書かない。
- **具体性**: 「コードを綺麗に書く」ではなく「2スペースインデントを使う」のように、検証可能な粒度で書く。
- **強調語**: `IMPORTANT` や `YOU MUST` を使うと遵守率が上がる。
- **育て方**: 最初から完璧を狙わず、Claudeが同じ間違いを繰り返した時・レビューで指摘された時に追記していく。Claudeに直接「これをCLAUDE.mdに追加して」と頼めば、その場で編集してもらえる。

## 5. 大きくなってきたら：`.claude/rules/`

- トピック別ファイル（`testing.md`, `security.md`など）に分割できる。
- frontmatterで`paths:`を指定すると、そのパターンに一致するファイルを扱っている時だけ読み込まれる「条件付きルール」にできる。
- 使い道の例：`gas-project`のようにサブディレクトリにCLAUDE.mdを作りたくない場合、`gas-project/.claude/rules/`に`paths: ["gas-voc-extract/**"]`のようなルールファイルを置けば、サブディレクトリにファイルを増やさずにプロジェクトごとの個別ルールを持たせられる。

## 6. `@path`によるインポート

- `See @README.md for project overview` のように書くと、そのファイルの内容がCLAUDE.md読み込み時に展開される。READMEやpackage.jsonの二重管理を避けられる。
- ただしインポートしたファイルもすべてコンテキストに載るため、行数・トークン削減にはならない（あくまで記述の整理目的）。

## 7. Auto Memory（自動メモリ）との違い

| | CLAUDE.md | Auto memory |
|---|---|---|
| 書く人 | 人間 | Claude自身 |
| 内容 | 指示・ルール | 気づき・パターン |
| 用途 | コーディング規約、ワークフロー | デバッグの知見、発見した癖 |

- 「常にpnpmを使って」のように口頭で伝えると、Claudeが自動でメモリ（プロジェクトのgitリポジトリに紐づく）に保存することがある。
- 恒久的なルールにしたいものは明示的にCLAUDE.mdに書く方が確実。
- `/memory`コマンドで、現在読み込まれているCLAUDE.md・ルール・自動メモリの内容を一覧・編集できる。

## 8. `gas-project`での実際の決定事項

- `gas-project`自体はGitリポジトリではなく、各サブディレクトリ（`gas-feedback-generate/`, `gas-voc-extract/`）がそれぞれ独立したGitリポジトリ・GASプロジェクトとしてclaspで管理されている、という前提を明記した。
- 共通ルールは `gas-project/.claude/CLAUDE.md` に配置（`./CLAUDE.md`ではなく`.claude/`配下を選択。整理上の理由）。
- 各GASプロジェクト固有のルールは、当面はサブディレクトリに個別のCLAUDE.mdを作らず、`gas-project/.claude/CLAUDE.md`のみで運用する方針。将来必要になれば、各サブディレクトリの`.claude/CLAUDE.md`として同じ配置パターンで追加する、または`.claude/rules/`のpaths指定で対応する。
