# Claude Desktop（Cowork）とCLI（ターミナル版Claude Code）の違い

- 学んだきっかけ: ターミナル版のClaude Code CLIを`gas-project`の作業用にMacへ試験導入する過程で、普段使っているCowork（デスクトップアプリ）との違いを整理した。

## 1. 全体像

- **Cowork**: Claude desktopアプリ内で完結。インストール不要、隔離されたサンドボックス（VM）内で動作するため安全性が高い。フォルダを指定してファイル整理・文書生成などのナレッジワークを任せる用途向け。
- **Claude Code（CLI）**: ターミナルにインストールして使う。実機に直接アクセスするため柔軟だが、リスク管理はユーザー側の責任になる。コーディング作業向け。
- 「Claude Code Desktop」という開発者向けの別アプリも存在し、Coworkとは別物。ただし両者は同じエンジンをベースにしており、CLAUDE.md・MCPサーバー・スキル・プラグイン・権限ルールなどの設定を共有できる。
- Cowork自体は以前は独立タブだったが、現在は「Home」タブに統合。メッセージ入力欄左下のセレクタで Chat / Cowork を切り替える。ChatとCoworkはプロジェクト・アーティファクトを共有している。

## 2. スケジューリングの違い（3階層）

Claude Codeには定期実行の仕組みが3種類あり、どれも「cron式」を使うが実体は異なる。

| | Cloud（Routines） | Desktop scheduled tasks | CLI（`/loop`） |
|---|---|---|---|
| 実行場所 | Anthropicのクラウド | 自分のPC | 自分のPC |
| PCの電源が必要か | 不要 | 必要 | 必要 |
| セッション（ターミナル）を開いておく必要 | 不要 | 不要 | **必要** |
| 再起動後も残るか | 残る | 残る | セッション終了で消える／7日で自動失効 |
| ローカルファイルへのアクセス | なし（クリーンな環境） | あり | あり |
| 最小間隔 | 1時間 | 1分 | 1分 |

- CLIの`/loop`はセッションスコープで、ターミナルを閉じると止まる・7日で自動失効するという点でむしろ最も"儚い"仕組み。Desktop scheduled tasksの方がアプリさえ入っていれば再起動後も持続する。

## 3. 「cronを設定できる」の正しい意味

- CLIの`/loop`もDesktopのscheduled tasksも、内部的に「cron式（書式）」を使って期限を管理しているだけで、**OS本来のcrontab/launchdには一切触れていない**。どちらもアプリ／セッション自身のポーリングで動く。
- 本物のOSレベルのcron/launchdを使うには、`claude -p`（非対話のheadlessモード）を**自分でcrontabやlaunchdに手動登録する**必要がある。これは実機で動くターミナル版のClaude Codeだからこそ可能（Bashツールで`crontab -e`相当のコマンドを実行させればよい）。
- Cowork（今使っているセッション）のシェルは隔離されたサンドボックス環境で動いているため、そこで`crontab`を実行してもサンドボックス内にしか反映されず、実機のcronには影響しない。

## 4. コーディングにおけるCLIの具体的なメリット

GUIがない場所・人の目を介さない自動化フローに組み込めることが本質。

- **gitフック**（pre-commit/pre-push）に`claude -p`を仕込み、コミット・プッシュ前に自動レビューを走らせる。
- **CI/CD（GitHub Actions）**に組み込み、PR作成やpushをトリガーに自動レビュー・自動修正を実行する。
- **リモートサーバー/SSH越しの作業**：GUIが使えない環境ではCLIが唯一の選択肢。
- **シェルスクリプトでの一括処理**：`for`ループで大量ファイルに同じ指示を適用するなど。
- **テスト・ビルド結果のパイプ渡し**：`npm test 2>&1 | claude -p "原因を直して"`のように他コマンドと連携できる。

Desktop版は「人がその場で対話しながら進める」用途、CLIは「人がいなくても・GUIがなくても動く」自動化パイプライン、という住み分け。

## 5. GitHub Actionsでの認証の考え方

- GitHub Actionsは基本的にGitHub側のクラウドVM（ホストランナー）でジョブを実行する。ローカルPCではない（セルフホストランナーを使う選択肢もある）。
- 認証は「今使っているログインセッションをそのまま渡す」のではなく、**同一アカウントから発行した別の鍵**をリポジトリのsecret（暗号化された環境変数）として登録する。
  - `ANTHROPIC_API_KEY`：Anthropic Consoleで発行。API従量課金として計上される。
  - `CLAUDE_CODE_OAUTH_TOKEN`：ローカルで`claude setup-token`を実行して発行。契約中のPro/Maxプラン内で消費される。
- イメージは「アプリパスワード」に近い。アカウント自体は同じだが、用途ごとに専用の鍵を発行し、後からその鍵だけを個別に無効化できるようにする仕組み。

## 6. MacへのCLIインストール手順（実施した内容）

- ネイティブインストーラ（推奨）: `curl -fsSL https://claude.ai/install.sh | bash`（バックグラウンドで自動更新される）
- Homebrew: `brew install --cask claude-code`
  - `--cask`はGUIアプリ用というだけの意味ではない。Anthropicが署名済みの**ビルド済みバイナリ**を配布しており、「ソースからビルドする通常のformula」ではなく「ビルド済み成果物をそのまま設置するcask」の仕組みで配っているため、CLIツールでもcaskになっている。
  - Homebrew版は自動更新されないため、`brew upgrade claude-code`で手動更新が必要。
- 確認コマンド: `claude --version`、`claude doctor`（設定・依存関係の診断）
- 起動・ログイン: 対象ディレクトリ（例: `gas-project/gas-feedback-generate/`のような各GASプロジェクトのサブディレクトリ）で`claude`を実行 → 初回はブラウザが開きログイン画面が表示される → 普段使っているアカウントでログイン。

## 7. 簡易疎通確認の手順

1. `claude --version` でインストール認識を確認
2. `claude doctor` で設定診断
3. テスト用ディレクトリ（例: `~/claude-test`）で`claude`を起動しログイン
4. 簡単なプロンプト（計算、または「hello.txtを作って中身を書いて」）を送り、応答とファイル操作の両方が正常に動くか確認
