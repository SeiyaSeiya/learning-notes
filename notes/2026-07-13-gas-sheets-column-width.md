# GASでスプレッドシートの列幅を確実に確保する方法（autoResizeColumnsの落とし穴）

- 学んだきっかけ: `gas-feedback-generate` プロジェクトで、保存先スプレッドシートの見出し・データが日本語文字化けならぬ「セル内に収まりきらず隠れる」問題が発生し、Claude Codeとの会話の中でその原因と対処法を調査した。

## 1. `autoResizeColumns` は日本語（全角文字）の幅推定が甘い

Google Sheets API（`SpreadsheetApp`）の `Range.autoResizeColumns` は、列内のテキストに合わせて自動で列幅を広げてくれる便利な関数だが、**全角文字（日本語）の実際の描画幅を過小評価する既知の癖がある**。

- 半角英数字前提の幅計算になっているらしく、日本語の見出し・データで呼んでも、実際の見た目より狭い列幅になり、文字が隠れたまま解消されないことがある。
- さらに厄介なのは、`autoResizeColumns` を「シート新規作成時の1回だけ」呼ぶような実装にしていると、**後から作られた別のシート（例: 月ごとのタブ）には反映されない**。テストで既に作成済みのシートは幅が補正されないまま残ってしまい、「直したはずなのに直っていない」という状態になりやすい。

## 2. 対処法: 文字列の表示幅から列幅（ピクセル数）を直接計算する

`autoResizeColumns` に頼らず、**文字列の表示幅を自前で計算し、`setColumnWidth` で確定的に列幅を設定する**方式に切り替えると解消できる。

- 全角文字は2、半角文字は1として1文字ずつ加算し、「表示幅の単位数」を出す（正規表現で「半角英数・半角カナ以外」を全角と判定する: `/[^\x01-\x7E\xA1-\xDF]/`）。
- 表示幅の単位数に「1単位あたりのピクセル数」を掛け、余白（パディング）を足して最終的なピクセル数を算出する。
- この関数をシートが新規作成でも既存でも**毎回**呼んで `setColumnWidth` で補正すれば、「後から作られたシートに反映されない」問題も解消できる（過去に別ロジックで作成され幅が補正されていないシートも、次にそのシートを開いた処理が走った時点で自動的に是正される）。

```js
const FULL_WIDTH_CHAR_PATTERN = /[^\x01-\x7E\xA1-\xDF]/;

function computeColumnWidthForText(text) {
  const PIXELS_PER_HALF_WIDTH_UNIT = 8;
  const PADDING_PIXELS = 20;

  let displayWidthUnits = 0;
  for (const char of text) {
    displayWidthUnits += FULL_WIDTH_CHAR_PATTERN.test(char) ? 2 : 1;
  }

  return PADDING_PIXELS + displayWidthUnits * PIXELS_PER_HALF_WIDTH_UNIT;
}
```

## 3. 見出し基準の幅だけでは足りない列がある（データ駆動での再調整）

見出し文字列の長さから列幅を決めても、**列によっては実データの方が見出しより長くなる**ことがあり（例: 長いタイトル、長い担当者名）、その場合はデータ側が隠れてしまう。

- 見出しだけを基準にするのではなく、「実データの方が長くなり得る列」をあらかじめ列挙しておき、バッチ書き込み完了後にその列だけシート内の全データ行（ヘッダー行を除く）を読み直して、最長の文字列に合わせて列幅を広げ直す、という2段構えの設計にした。
- 1行書き込むたびに毎回全行スキャンすると無駄なので、「1バッチの書き込みが完了したタイミングで1回だけ」実行するようにし、パフォーマンスとのバランスを取った。
- 日付列など、値がDateオブジェクトのまま渡ってくる場合は、幅計算の前に文字列へ正規化してから文字数を数える必要がある点も注意（Dateオブジェクトのまま `String()` すると想定外の長い文字列になり得る）。

## 4. まとめ: GASでのセル幅調整の教訓

- `autoResizeColumns` は「それっぽく動くが日本語では信頼できない」典型例。日本語をメインで扱うシートでは、文字列長からの決定的な計算＋`setColumnWidth` の方が確実。
- 「初回だけ実行する」設計は、後から増える対象（今回で言えば月ごとに増えていくシート）に対して効かなくなる典型的な落とし穴。冪等に「毎回補正する」設計にしておくと、この手の見落としを避けやすい。
- 見出し基準の幅とデータ基準の幅は本質的に別問題であり、両方をカバーするには「見出し用の一律計算」と「データ実測に基づく再調整」の2段階が必要になる。
