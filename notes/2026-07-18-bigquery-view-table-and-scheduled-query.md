# BigQueryでビュー・テーブルを作成し、スケジュールクエリ（トリガー）で定期反映する方法

## 概要
- **学んだきっかけ**: 社内のBigQueryプロジェクトで、既存データを加工した新しいビューを作成し、その結果を毎日決まった時刻に別のテーブルへスナップショットとして反映する仕組みを構築した際の一連の作業。
- ビュー・テーブルのUI上での作成手順から、パーティショニング／クラスタリングの理解、スケジュールクエリ（BigQueryにおける「トリガー」に相当する仕組み）の設定方法までをまとめる。

## 1. ビューとテーブルの違い

ビューは実データを持たない。参照されるたびに、ビューの定義SQLがその場で展開・実行される「保存されたクエリ」のようなものである。そのため、ビュー単体を定期実行してデータを蓄積するという概念は存在しない。データを物理的に保持したい場合は、別途テーブルを用意し、そこにクエリ結果を書き込む必要がある。

この性質から、「ビューの定義SQLを定期実行するトリガー」と「ビューの結果をテーブルに反映するトリガー」は本来別々の処理に見えるが、実際には後者のトリガー（テーブルへの書き込みクエリ）を実行するだけで、その中でビューの定義SQLも自動的に評価されるため、1つのトリガーで両方を兼ねられる。

## 2. UI上でのビュー作成手順

1. クエリエディタでビューの定義となるSELECT文を記述する
2. エディタ右上の「保存」ドロップダウンから「ビューを保存」を選択
3. ダイアログでプロジェクト／データセット／ビュー名を指定して保存

ビューはこの方法で作成するのが基本であり、テーブル作成ダイアログのような専用のGUIウィザードは無い（SQLを書いて保存する形）。

## 3. UI上でのテーブル作成手順

1. Explorerパネルでデータセットの「⋮」メニューから「テーブルを作成」を選択
2. 「ソース」で作成方法を選ぶ。主な選択肢は以下の通り。
   - 空のテーブル：スキーマを手入力して空のテーブルを作る
   - 既存のテーブル/ビュー：指定した既存テーブル/ビューのスキーマとデータを1回だけコピーする（自動更新はされない、単発のコピー）
   - Google Cloud Storage／アップロード／ドライブなど：外部ファイルから読み込む
3. スキーマ、パーティショニング、クラスタリングなどの詳細設定を行い「テーブルを作成」

### Apache Iceberg用テーブルのチェックボックス

「Create a BigQuery table for Apache Iceberg」というチェックボックスがある。これは、データをBigQuery内部ストレージではなく、自分のCloud Storageバケット上にApache Iceberg形式（オープンなテーブルフォーマット）で保存しつつ、BigQueryからはネイティブテーブルと同様にSQLで読み書きできるようにする機能。Spark等の外部エンジンとデータを共有したい場合に使う。通常のBigQuery運用のみであれば不要。

## 4. パーティショニング

テーブル内のデータを特定の列（多くは日付・タイムスタンプ）の値ごとに物理的に分割して保存する設定。クエリ時に該当パーティションだけを読み込むことで、スキャン量（コスト）と速度が改善する。

選択肢は主に3つ。

- パーティショニングなし
- 取り込み時間により分割：データがロードされた日時を基準に自動分割
- フィールドにより分割：DATE/TIMESTAMP/INTEGER型の列を指定し、その値で分割

範囲指定で絞り込むことが多い低〜中カーディナリティの列（代表例：日付）に向いている。

## 5. クラスタリング

指定した列（最大4列）の値順にデータを並び替えて保存する設定。パーティショニングが「テーブルを物理的な塊に分ける」のに対し、クラスタリングは「各パーティション内（またはテーブル全体）でのデータの並び順」を制御する。

WHERE句やGROUP BY句で頻繁に使う、値の種類が多い（高カーディナリティな）列に向いている。指定順が優先度になり、1列目が最も効果を発揮する。

## 6. 既存テーブルのパーティショニング／クラスタリング設定の確認方法

- UIから確認：テーブルを開き「詳細」タブの「テーブル情報」欄を見る。パーティショニング・クラスタリングが設定されている場合のみ、それぞれの項目が表示される（未設定の場合は項目自体が表示されない）
- SQLから確認：

```sql
SELECT ddl
FROM `プロジェクト.データセット.INFORMATION_SCHEMA.TABLES`
WHERE table_name = 'テーブル名';
```

DDLに`PARTITION BY`や`CLUSTER BY`句が含まれていなければ未設定と判断できる。

## 7. スケジュールクエリ（トリガー）

BigQueryには従来型DBのような「トリガー」機能は無く、定期実行にはBigQuery Data Transfer Serviceが提供する「スケジュールクエリ」を使う。

### 設定できる3要素と設定場所

- 元テーブル／ビュー：クエリ文字列のFROM句で指定
- 処理（抽出・加工ロジック）：クエリ文字列そのもの
- 保存先テーブル：クエリの書き方によって設定場所が変わる
  - SQLが単純なSELECTのみの場合：スケジュールクエリの設定画面にある「クエリ結果の宛先」欄（データセット／テーブル名／書き込み設定）で指定
  - SQLにCREATE OR REPLACE TABLE AS SELECTやINSERT INTO、MERGEなどのDDL/DMLを使う場合：SQL文自体に保存先が書かれるため、設定画面の宛先欄は不要

### 作成手順

1. クエリエディタでSELECT文（または宛先を含むDDL/DML）を記述して実行
2. 「スケジュール」→「新しいスケジュールクエリを作成」
3. 繰り返し頻度（毎日／毎週など）と実行時刻を設定。タイムゾーンの指定に注意（選択できない場合はUTC基準で時刻を調整する）
4. SELECTのみの場合は「クエリ結果の宛先」でデータセット／テーブル名／書き込み設定（追加・上書き・書き込みしない）を指定
5. 実行に使うサービスアカウントに、参照先テーブル・ビューすべてへの読み取り権限があることを確認して保存

### 書き込み設定（Write disposition）

- 追加（WRITE_APPEND）：既存データに追記。日々の履歴を蓄積したい場合に使う
- 上書き（WRITE_TRUNCATE）：毎回全データを洗い替え。最新スナップショットのみを保持したい場合に使う
- 書き込みしない（WRITE_EMPTY）：テーブルにデータが無い場合のみ書き込む

### 権限に関する注意点

スケジュールクエリの元がビューの場合、ビューは実行時に定義SQLへ展開されるため、ビューを実行するアカウントには、ビュー自体だけでなく、ビューの定義SQLが参照している元テーブルすべてへの読み取り権限が必要になる（BigQueryの「承認済みビュー」を設定していない場合）。

## 8. 日付ごとに分割されたテーブル（ワイルドカードテーブル）の扱い

パーティションテーブルとは別に、日付ごとに物理的に別テーブルとして分割されている構成（例：`dataset.table_20260718`のような命名）もある。この場合はワイルドカード指定と`_TABLE_SUFFIX`で対象日を絞り込む。

```sql
SELECT *
FROM `プロジェクト.データセット.テーブル名_*`
WHERE _TABLE_SUFFIX = FORMAT_DATE('%Y%m%d', DATE_SUB(CURRENT_DATE('Asia/Tokyo'), INTERVAL 1 DAY))
```

パーティションテーブルの`WHERE d = DATE_SUB(...)`とは書き方が異なる点に注意。

## 9. 複数ソースからのフォールバック取得（COALESCE）

同じ意味のカラムが複数のテーブルに分散している場合（例：新しいテーブルに移行中で、一部データは新テーブルに無く旧テーブルにしか無いなど）、`COALESCE`で優先順位をつけて1つの値に統合できる。

```sql
COALESCE(A.column_x, B.column_x) AS column_x
```

Aの値がNULLの場合のみBの値を採用する、というシンプルなフォールバックロジックとして活用できる。

## 10. 全体の設計パターンまとめ

今回構築した仕組みは、以下の3層構造になっている。

1. 生データを保持する複数の元テーブル（パーティションテーブルや日付分割テーブルなど、形式は様々）
2. それらを統合・加工して1つの論理的なテーブルとして見せる「ビュー」（実データは持たない）
3. ビューの結果を毎日決まった時刻にスナップショットとして書き込む「物理テーブル」（スケジュールクエリによって定期更新）

この構成により、加工ロジックの変更はビューのSQLを直すだけで完結し、かつ最終利用者（BIツールなど）は重い加工クエリを毎回実行せず、軽量な物理テーブルを参照するだけで済むという利点がある。
