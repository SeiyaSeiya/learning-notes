# BigQueryの分析用テーブル（listテーブル）に対するテストSQL作成の観点整理

## 概要
- **学んだきっかけ**: warehouse側の基盤テーブルを元に分析用テーブル（以下「テーブルA」）を作成した際、過去に別の分析用テーブル（以下「テーブルB」）で実施したテストの依頼プロンプトを参考にしながら、テーブルAに対するテスト観点と、テストSQLを書いてもらうための依頼プロンプトを整理した。
- 「分析用テーブルが仕様通り正しく作られているか」を機械的に検証するための観点は、大きく4つに分類できることと、それぞれに見落としがちな落とし穴があることを学んだ。
- ※実際のテーブル名・カラム名は社内の秘匿情報のため、本メモでは「テーブルA」「テーブルB」のように抽象化して記載する。

## 1. 背景：なぜテスト観点を先に整理するか

warehouseの生データをJOIN・CASE・COALESCEなどで加工して作る分析用テーブル（list側テーブル）は、目視チェックだけだと見逃しやすい落とし穴が多い（件数の相殺、シャード欠損、分岐ロジックの誤りなど）。そのため、テストSQLを誰か（別セッションのClaudeや同僚）に書いてもらう前に、まず自分でテスト観点を整理し、依頼プロンプトの形にしておくと、依頼内容の抜け漏れが減り、出てきたテストSQLの妥当性も判断しやすくなる。

## 2. テスト観点は4つに分類できる

- カラム名・型チェック
- 行数チェック
- データ存在チェック
- データ内容チェック

一見シンプルだが、単純に「合計件数が一致すればOK」のように捉えると不十分で、それぞれに注意点がある。

## 3. カラム名・型チェック

list側テーブルAのカラム名・データ型が仕様書通りであることを、`INFORMATION_SCHEMA.COLUMNS`と仕様書を突き合わせて確認する。すべてのチェックの前提となる、最も基本的な観点。

```sql
SELECT column_name, data_type
FROM `プロジェクト.データセット.INFORMATION_SCHEMA.COLUMNS`
WHERE table_name = 'テーブル名'
ORDER BY ordinal_position;
```

## 4. 行数チェック（合計件数だけでは不十分）

warehouse側テーブルAとlist側テーブルAの合計`COUNT(*)`を比較するだけだと、「行の漏れ」と「JOINによるファンアウト（重複）」が同時に起きた場合、誤差が相殺されて一致してしまい見逃す可能性がある（例：10行漏れて10行重複、合計は一致するが中身は破綻）。

そのため、主キー単位で以下の2つを確認する必要がある。

- warehouse側とlist側で、キーの集合が過不足なく一致すること（`EXCEPT DISTINCT`などで差分を検出）
- list側でキーが重複していないこと（結合先テーブルのキー重複によるファンアウトが起きていないか）

## 5. データ存在チェック（2種類ある）

「存在チェック」は一括りにすると曖昧になりやすく、原因も対処も異なる2種類に分けて考えるとよい。

1. **シャード欠損のカナリアチェック**：日付シャード方式（`_TABLE_SUFFIX`で対象日を指定するワイルドカードテーブル）のソーステーブルは、対象日のシャードが生成されていない場合、エラーにならず単に0件を返す。その結果JOIN先が全てNULLになるが、これは「該当データが無いので正常なNULL」なのか「シャード自体が欠損した異常」なのか、行単位の比較だけでは区別できない。事前に「対象日のシャードに1件以上データが存在するか」を確認しておく必要がある。
2. **JOIN失敗による意図しないNULLの検出**：warehouse側に外部キー（マスタ参照用のID列）が存在するにもかかわらず、結合先のマスタテーブルにそのキーが無くJOIN結果がNULLになっている行がないかを確認する。

## 6. データ内容チェック（再実装による突合が必要）

CASE文やCOALESCEによる変換ロジックは、件数一致やJOIN成功だけでは正しさを検証できない。分岐先の値が間違っていても、結果は非NULLで件数も変わらないため、行数チェック・存在チェックをすべて通過してしまう。

これを防ぐには、テストSQL自体が変換ロジックを独立に再実装し、期待値と実際の出力値を1行ずつ突き合わせる方式が必要。対象は例えば以下のようなもの。

- 区分フラグをテキストに変換するCASE文（全分岐を網羅）
- 複数カラムの組み合わせで分岐するステータス系カラム（例：公開フラグ×公開日時の組み合わせ）
- `COALESCE(warehouse側, source側)`のような優先順位ロジック（warehouse側がNULLの行に絞って、source側の値が採用されているかを確認）
- タイムゾーン変換（UTC→Asia/Tokyo）

また、対象日の実データに全ての分岐パターンが存在するとは限らないため、実データに存在しない分岐は「検証対象外」であることをテスト結果に明記しておくとよい。

## 7. 添付すべき資料は5点、うち実装SQLも必要

テストSQLを誰かに依頼する際、当初は仕様書と各テーブルのスキーマだけで足りると考えていたが、それだけでは「JOINのキーが何か」「COALESCEでどちらを優先しているか」をデータから逆算しなければならず非効率だと気づいた。実装SQL（今回作成したlist側テーブルAの定義SQLそのもの）も、JOIN構造やロジックを理解するための参照資料として渡す必要がある。

ただし役割は仕様書と異なる点に注意。仕様書は「あるべき姿」、定義SQLは「実際の実装」であり、正誤判定の基準はあくまで仕様書側に置く（定義SQLは実装を鵜呑みにしてなぞるための資料ではない）。

最終的な添付資料は以下の5点。

1. 仕様書（あるべきカラム定義・変換ルール）
2. 分析用テーブルの定義SQL（実装、JOINキー・ロジック理解の参照用）
3. 分析用テーブルの実スキーマ（`INFORMATION_SCHEMA.COLUMNS`で機械的に取得可能）
4. 元となるwarehouseテーブルの実スキーマ
5. 結合先テーブル群の実スキーマ

## 8. 依頼プロンプトのテンプレート

```
データ分析テーブルの正しさのチェックを行いたいです

warehouse.○○（＋結合先テーブル群）をベースに、
list.○○を作成しました

確認したいことは以下の4観点です

①カラム名・型チェック
list.○○のカラム名・データ型が仕様通りであること（添付-1:仕様書）

②行数チェック
・warehouse.○○とlist.○○の合計件数が一致すること
・さらに、主キー単位でwarehouse側とlist側の集合が過不足なく一致すること（EXCEPT DISTINCTなどで差分を検出）
・list側で主キーが重複していないこと（結合先とのJOINでファンアウトが起きていないこと）
　※合計件数の一致だけでは「漏れ」と「重複」が相殺して見逃される可能性があるため、集合単位・重複チェックを必須とする

③データ存在チェック
・日付シャード方式のソーステーブルについて、対象日のシャードにそもそも1件以上データが存在すること（シャード欠損がないかのカナリアチェック）
・warehouse側でキーが存在するにもかかわらず、結合先マスタとのJOIN結果がNULLになっている行がないこと

④データ内容チェック
list.○○で変換・加工しているカラムについて、変換ロジックをテストSQL側で再実装し、期待値と実際の値を突き合わせること
　※対象日のデータに全分岐パターンが実在するとは限らないため、実データに存在しない分岐は「検証対象外」である旨をテスト結果に明記すること

○○の定義SQL（添付-2）
list.○○(添付-3）
warehouse.○○（添付-4）
結合先テーブル群のスキーマ（添付-5）

※定義SQLはJOINキーや変換ロジックを理解するための参照用です。正誤判定の基準はあくまで添付-1の仕様書とします

チェックSQLを書くことを依頼したく、最終的にはそのSQLの結果を見ればOKなのか、NGなのかが一目でわかるようにしたいです　〇×で判別でき、×が1個でもあればアウトという考え方です

・④は①〜③の正しさが前提になるので、まずは①〜③について検証を行い、その後④をチェックするという流れがよいかと思います
```

## 9. テーブルBとの比較から得た学び

以前テーブルBをテストした際のプロンプトと比較すると、テーブルの設計パターンによってテストの重点が変わることがわかった。

- テーブルBのwarehouse CTEには`QUALIFY ROW_NUMBER() OVER(PARTITION BY ... ORDER BY 通し番号カラム DESC) = 1`という重複排除ロジックがあった。この場合、行数チェックは「warehouseの生カウントとの一致」では成立せず、QUALIFYのロジックをテストSQL側でも再現した上での件数比較が必要になる。テーブルAにはこのような重複排除ロジックが無かったため、単純な件数比較で済んだ。
- テーブルBは複数の集計用CTE（GROUP BYで主キー単位に集約したCTE）を挟んでからJOINすることで、ファンアウトを設計時点で防ぐ工夫がされていた。テーブルAは集約を挟まない直接JOINが多く、その分キーの一意性チェック（行数チェックの一部）がより重要になる。

新しい分析用テーブルをテストする際は、まず「重複排除ロジック（QUALIFY等）の有無」「集計CTEを挟んでファンアウトを防ぐ設計になっているか」を確認してから、4つのテスト観点のうちどこに重点を置くかを判断する、という進め方が今後も使えそうだと感じた。
