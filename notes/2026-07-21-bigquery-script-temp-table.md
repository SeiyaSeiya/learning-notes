# BigQueryのスクリプト機能（DECLARE / CREATE TEMP TABLE / IF文）の基礎

## 概要

- **学んだきっかけ**: 分析用テーブルの検証用チェックSQL（関連メモ: [2026-07-21: BigQueryの分析用テーブルに対するテストSQL作成の観点整理](2026-07-21-bigquery-list-table-testing-checklist.md)）を実行してみたところ`CREATE TEMP TABLE`や`INSERT`が動いているのが見え、「本番テーブルに影響しないか」「なぜ事前にバイト数（処理量）の見積もりが出ないのか」が気になったので、BigQueryの「スクリプト」機能について整理した。
- 要点は3つ。①`CREATE TEMP TABLE`は一時的な入れ物であり本番データセットには一切影響しない、②スクリプトは単一SELECTとは実行のされ方が違う、③そのため事前のバイト数見積もりが効かないケースがある。

## 1. そもそも「スクリプト」とは何か

BigQueryの「スクリプト」は、複数のSQLステートメントをセミコロン区切りで並べて、1つのジョブとしてまとめて実行できる仕組み。普段書く1本のSELECT文とは違い、以下のようなものを組み合わせられる。

- `DECLARE` / `SET`：変数の宣言と代入
- `CREATE TEMP TABLE`：一時テーブルの作成
- `IF` / `LOOP` / `WHILE`：制御構文
- 複数の`INSERT` / `CREATE` / `SELECT`を順番に実行

実行すると、ステートメントごとに個別の「子ジョブ」としてBigQuery上で処理され、コンソールの実行結果画面から後で1文ずつの結果や処理バイト数を確認できる。

## 2. CREATE TEMP TABLE：一時テーブルのライフサイクル

- 一時テーブルは、そのスクリプト（またはインタラクティブなセッション）の中だけに存在するテーブルで、データセットを指定せずに作る。
- **ライフサイクル**：スクリプトの実行が終わると同時に自動的に破棄される。データセットの一覧に表示されることはなく、他のクエリや他のユーザーから参照することもできない。
- **本番テーブルへの影響**：`CREATE TEMP TABLE`は新しくデータセット領域を作るわけではないので、既存の本番データセット・本番テーブルには一切影響しない。読み込み元のテーブル（例：`warehouse.exams`など）に対してはSELECT（読み取り）しか発生していない。
- `INSERT INTO`も、書き込み先が一時テーブルである限り、本番テーブルへの書き込みにはならない。
- **料金**：一時テーブルの保存自体には追加のストレージ課金は発生しない。かかるのは通常のクエリと同じ「スキャンしたバイト数」に対するクエリ課金のみ。
- **セッションモードとの違い**：BigQueryコンソールで明示的に「セッションを開始」して使う場合、一時テーブルはセッション全体（最大24時間）で使い回せる。一方、今回のような1回実行のスクリプトでは、そのジョブの実行中だけ存在し、実行完了と同時に消える。手動でのクリーンアップは不要。

## 3. DECLARE：スクリプト内変数

スクリプトの先頭で変数を宣言し、`DEFAULT`で初期値を与えられる。

```sql
DECLARE target_date DATE DEFAULT DATE_SUB(CURRENT_DATE('Asia/Tokyo'), INTERVAL 1 DAY);
DECLARE target_date_suffix STRING DEFAULT FORMAT_DATE('%Y%m%d', target_date);
```

注意点として、変数はSQL文中の「値」としては使えるが、テーブル名やデータセット名などの「識別子」の位置には直接使えない。対象日を変えるだけで動的にテーブル名を組み立てたい、というような場合は`EXECUTE IMMEDIATE`で文字列としてSQLを組み立てて実行する必要がある（今回のスクリプトでは、日付は`WHERE d = target_date`や`_TABLE_SUFFIX = target_date_suffix`のような「値」としてのみ使っているので、この制約には引っかかっていない）。

## 4. IF ... THEN ... ELSE ... END IF：条件分岐

条件によって、実行するステートメント群をブロックごと分岐できる。

```sql
IF EXISTS (SELECT 1 FROM check_results WHERE judge = '×') THEN
  -- 前段のチェックでNGがあった場合の処理
  INSERT INTO check_results VALUES (...);
ELSE
  -- 無かった場合の処理（本来やりたかったチェックを実行）
  INSERT INTO check_results SELECT ...;
END IF;
```

BigQueryの`IF`には実は2つの顔があり、混同しやすいので注意。

- **式としてのIF関数**：`IF(条件, 真の場合の値, 偽の場合の値)` のように、SELECT句の中で値を返す関数。
- **制御構文としてのIF文**：`IF 条件 THEN ... END IF;` のように、実行するステートメントの塊を分岐させる文。

見分け方は、`IF`の直後が`(`（カッコ）で式が続くか、条件式のあとに`THEN`が来るか。

## 5. なぜ実行前にバイト数を見積もれないのか

単一のSELECT文なら、実行前に「このクエリを実行すると◯◯バイトが処理されます」という見積もり（ドライラン）が出るのが普通だが、スクリプト形式にすると出ないことがある。理由は大きく2つ。

1. **スクリプト全体の構造による**：IF分岐がある場合、実際にどちらのブロックが実行されるかは実行時（前段の判定結果）にならないと確定しない。ドライランは静的な解析なので、分岐先まで正確に見積もれない。
2. **ワイルドカードテーブル＋変数によるシャード絞り込みができない**：`_TABLE_SUFFIX = 変数名`のように、日付シャード方式のテーブル（`source.exams_*`など）への絞り込み条件がDECLAREした変数になっている場合、解析の時点では変数の中身が確定していないため、どのシャードだけを読めばよいかを事前にプルーニングできない。

**対処法**：正確な処理量を知りたい場合は、実行後に該当ジョブの詳細（BigQueryコンソールの「ジョブ情報」、または`INFORMATION_SCHEMA.JOBS_BY_PROJECT`の`total_bytes_processed`列）を見れば確認できる。どうしても事前に見積もりたい場合は、変数の部分を一時的にリテラル文字列（例：`'20260720'`）に置き換えて試すとよい。

## 6. 実務での使いどころ・注意点

- 一時テーブルを使うと、「複数のチェック観点を順番に検証し、1つの結果テーブルに集約する」といったテストSQLが1本のスクリプトで完結できる。一時テーブルを使わない場合、観点ごとに別々のクエリを実行して結果を手動で見比べる必要がある。
- ただし一時テーブルはあくまで「その場限り」。チェック結果を後から見返したい・他のクエリから参照したい場合は、`CREATE TEMP TABLE`を`CREATE TABLE `プロジェクト.データセット.テーブル名``のように、通常のテーブルとして永続化する必要がある。
- スクリプトの途中でエラーが起きても、それまでに実行された`INSERT`などの結果は自動的にロールバックされない（BigQueryのスクリプトはトランザクションではない）。ただし書き込み先が一時テーブルであれば実害はなく、再実行すればまっさらな状態からやり直せる。
