# アンケートVOC抽出作業とopenpyxlの基礎、業務効率化アプローチの考察

- 学んだきっかけ: カスタマーサクセス業務で、サービス利用者アンケートの自由記述・集計結果から「VOC（顧客の声）」を抽出し、既存の改修要望リスト（Excel）の形式に整理する作業をClaudeと一緒に行った。その過程でExcel操作用ライブラリ`openpyxl`の基本と、今後同種の事務作業を効率化する方法について学んだ内容をまとめる。

## 1. VOC抽出作業の全体的な流れ

- 入力は2種類のExcelファイル：①アンケートの生回答＋集計（「フォームの回答」タブと、それを要約した「まとめ」タブ）、②既存の改修要望リスト（ステータス／発生日／概要／対象プロダクト／関連機能／ユーザー／詳細／背景／優先順位、という決まった列を持つ）。
- 「まとめ」タブには、あらかじめ「優先度／カテゴリ／主な改善要望／原文」という形で自由記述が要約された一覧表があり、これを抽出のベースにするのが効率的だった（生の自由記述全件を毎回読み直すのではなく、既に要約済みのタブを起点にする）。
- 出力先の列に一つずつ値を当てはめる際、「概要」「詳細」「背景」はまとめタブの記述をベースに要約・言い換えが必要になるため、この部分は人（Claude）の解釈が必要で、単純な転記だけでは終わらない作業だと分かった。

## 2. アンケートの「設問設計」からユーザー属性を逆算する考え方

- アンケートの回答者属性（役職など）ごとに、回答している設問と回答していない設問がある、という構造に気づいたことが今回のポイントだった。
- 特定の役職しか回答していない設問（例：部下の管理・進捗確認に関する設問）から出てきた要望は「管理者向け」、全回答者が回答する設問（例：コンテンツ全般の使い勝手）から出てきた要望は「利用者・管理者どちらにも共通」と判定できる。
- この「誰が回答しているか」を先に確認しておくと、抽出後の「ユーザー」列（対象者）を後から個別に推測するより、根拠を持って機械的に埋められる。今後同種のアンケート分析をする際にも使える視点。

## 3. openpyxl（Excel操作ライブラリ）の基礎

Excelアプリを開かずに、Pythonから`.xlsx`ファイルを直接読み書きできるライブラリ。既存ファイルの書式・数式・入力規則（ドロップダウン等）を壊さずに特定セルだけ埋める、という用途に向いている。

```python
import openpyxl
from openpyxl.styles import Font, Alignment
from datetime import datetime

# 既存ファイルを開く（data_only=Falseだと数式はそのまま文字列として保持される）
wb = openpyxl.load_workbook("path/to/file.xlsx", data_only=False)
ws = wb["シート名"]

# セルの読み書き（row, columnは1始まり）
ws.cell(row=2, column=1, value="検討前")
ws.cell(row=2, column=2, value=datetime(2026, 7, 12)).number_format = "yyyy-mm-dd"

# 書式（フォント・折り返し）
cell = ws.cell(row=2, column=3, value="長文の詳細説明...")
cell.font = Font(bold=True)
cell.alignment = Alignment(wrapText=True, vertical="top")

# 複数セルをまとめて読む
for row in ws.iter_rows(min_row=1, max_row=10, values_only=True):
    print(row)

wb.save("path/to/file.xlsx")
```

- 新規ファイルを作る場合は`openpyxl.Workbook()`、既存ファイルは`load_workbook()`。
- 入力規則（ドロップダウンリストなど）は`ws.data_validations.dataValidation`、結合セルは`ws.merged_cells.ranges`で事前に確認できる。既存の入力規則の範囲外に書き込まないよう、事前確認してから値を入れるとよい。
- pandasでDataFrame経由で書き出すと、既存ファイルの書式・数式・入力規則が失われることがあるため、「既存ファイルの体裁を保ったまま一部セルだけ埋める」用途ではopenpyxlで直接セルを操作する方が安全。

## 4. 作業環境特有の注意点（ファイル権限）

- Claude Desktop（Cowork）環境では、アップロードされたファイルは読み取り専用の権限でコピーされることがあり、そのままの権限で別ディレクトリにコピーしても書き込み時に`PermissionError`になることがあった。
- 対処法は、作業用にコピーしたファイルに対して`chmod`で書き込み権限を付与してから`openpyxl`で保存する、という一手間。ファイルコピー→権限確認→書き込み、の順番を意識するとよい。
- 完成物を最終的な保存先フォルダに置く前に、一時作業用のディレクトリで組み立てて、最後に確定版だけコピーする、という2段階の進め方が安全（保存先フォルダのファイルは一度書き込むと上書き・削除がしづらい制約があるため）。

## 5. 業務効率化のアプローチ：プロンプト化 vs スキル化

同じような事務作業を繰り返す場合の効率化手段として、「再利用プロンプト」と「スキル化」の2つを比較検討した。

- **再利用プロンプト**：作業の判断基準（列のマッピングルール、ユーザー属性の判定方法、出力フォーマットなど）を一つの指示文としてまとめておき、次回以降はそれをそのまま貼り付けて依頼する方式。作成のハードルが低く、すぐに使い始められる。
- **スキル化**：同じ内容をより体系立てたドキュメント（SKILL.md）としてまとめ、決まった呼び出し方で毎回実行できるようにする方式。再現性・呼び出しやすさは上がるが、正式に登録する手間がかかる。
- 使い分けの考え方：「判断・解釈が伴う作業（曖昧な自由記述を一定の基準で要約・分類するなど）」は、固定スクリプトだけで完全自動化するのが難しく、プロンプト化して都度AIに判断させる方が向いている。逆に「毎回まったく同じ形式のデータを、判断なしにただ書き写すだけ」の作業であれば、プロンプトよりも固定スクリプト（マクロ等）の方が速く確実。
- プロンプト化は「毎回ゼロから作業方針を説明する手間」を減らすものであって、「結果をチェックする手間」を無くすものではない、という点は認識しておく必要がある。効率化の効果はあっても、最終レビューは引き続き必要。
- 運用が安定し、頻度が増えてきたタイミングで、プロンプトをベースにスキル化するとさらに手間が減る、という段階的な進め方が良さそう。

## 6. まとめ

- 「既に要約されたデータ（まとめタブ）を起点にする」「アンケートの設問設計からユーザー属性を逆算する」など、VOC抽出のような解釈を伴うデータ整理作業には、抽出元の構造を先に理解しておくことが精度を上げる近道になる。
- openpyxlは、既存Excelファイルの書式・数式・入力規則を壊さずに特定セルだけを更新する用途に強い。pandas経由の書き出しとは使い分けが必要。
- 繰り返し作業の効率化は、作業の性質（判断が伴うか、単純転記か）によって「プロンプト化」と「固定スクリプト化／スキル化」を使い分けるのが合理的。
