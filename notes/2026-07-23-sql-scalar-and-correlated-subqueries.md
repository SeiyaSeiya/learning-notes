# SQL: SELECT句内のサブクエリ（スカラ・サブクエリと相関サブクエリ）の挙動

## 概要
SELECT句にサブクエリ（`SELECT`文）を埋め込んだ際の、クエリ全体の出力件数の決まり方と、別々のカラムとして定義したサブクエリが最終結果でどのように突合（紐づけ）されるのかについてのまとめ。

## 1. 全体の件数はどうなるか（スカラ・サブクエリ）
**結論: メインとなる外側のSQL（`FROM` や `GROUP BY`）の件数に完全に一致する。**

* **理由**: 
  SELECT句内のサブクエリは「**スカラ・サブクエリ（Scalar Subquery）**」と呼ばれ、「単一の値を計算して取得する」ための関数のような動作をする。テーブル同士を結合する `JOIN` とは異なり、行数を増やしたり減らしたりといった全体の件数への影響は一切与えない。
* **注意点**:
  * **複数件ヒットした場合**: サブクエリが複数件のレコードを返してしまうとエラー（`Scalar subquery produced more than one element`）になる。
  * **0件の場合**: 該当するデータが存在しない場合はエラーにはならず、そのカラムの値が `NULL` となる。

## 2. カラムの突合（紐づけ）の仕組み（相関サブクエリ）
**結論: `WHERE` 句での条件指定により、外側の行と1行ずつ照合されている。**

* **仕組み**:
  サブクエリ内に `WHERE id = ap.client_id` のような結合条件を含める仕組みを「**相関サブクエリ（Correlated Subquery）**」と呼ぶ。
  データベース内部では、「外側の行を1行ずつ処理しながら、その行の `client_id` の値を使ってサブクエリの条件に当てはめ、毎回検索を実行する」という動きをしている。
* **イメージ（Excelやプログラミングでの例え）**:
  * **Excel**: `VLOOKUP` 関数で、外側の値を検索キーにして別テーブルからピンポイントで値を取得する動きと同じ。
  * **プログラミング**: `for` ループで外側の行を1行ずつ回しながら、内部で該当するデータを検索してプロパティにセットしていく動きと同じ。

このように、外側のテーブルの値を検索キーとして都度検索を行っているため、カラムが別々に記述されていても、最終結果の行の中で値がピタッと綺麗に紐づいて出力される。
