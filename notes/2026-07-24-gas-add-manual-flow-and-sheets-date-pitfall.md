# GASの既存プロジェクトに手動実行フローを追加する勘所（Sheetsの日付自動変換・関数流用・列順整合）

- 学んだきっかけ: `gas-voc-extract`（MTG文字起こしからGeminiでVOC抽出→Slack自動投稿するGASプロジェクト）に、別系統の「顧客アンケート集約VOCをSlackへ一括投稿する手動実行フロー（`ManualVocPost.js`）」を追加する作業をClaude Codeと進めた。その過程で踏んだ落とし穴と設計判断を整理する。

## 1. 既存プロジェクトへの機能追加は「グローバルスコープ共有」を活かして関数を流用する

GASは同一プロジェクト内の全 `.js` ファイルが実行時に1つのグローバルスコープへ結合される。別プロジェクトへコードを複製せず、同じプロジェクトにファイルを1枚足すだけで、既存の `postToSlack` / `formatVocForSlack` などをそのまま呼べる。

- 今回は新フロー専用に別プロジェクトを作らず、`ManualVocPost.js` を1枚追加して既存のSlack投稿関数を再利用した。
- 手動実行フローなので**トリガーは設定せず**、GASエディタから対象関数を選んで実行する運用にした。処理を「①整形して中間シートへ書き出す関数」と「②中間シートを読んでSlack投稿する関数」の2つに分けると、①の出力を人が目視確認・修正してから②を回せる（AIや自動整形の結果を人がレビューできる余地を残す）。

## 2. 既存関数へ引数を足すときは「後方互換のデフォルト」で既存挙動を壊さない

既存の `postToSlack(text, name)` は投稿本文の先頭に「自動抽出されました＋@メンション」の固定文言を必ず付けていた。新フローではこの文言が不適切だったが、既存の自動投稿フローの挙動は1ミリも変えたくない。

- `postToSlack(text, name, headerText)` と**第3引数を追加**し、`headerText` 未指定時は従来の文言をデフォルトとして使い、指定時のみ差し替える形にした。
- こうすると既存の呼び出し箇所（引数2つ）は一切変更不要で挙動も不変、新フローだけが第3引数で文言を上書きできる。「共有スコープで既存を壊さず拡張する」定石。

## 3. 【最重要】Sheetsは"日付に見える文字列"を書き込むと勝手にDate型へ変換する

今回いちばんハマった不具合。整形関数が発生日を `"2026/07/12"` という**文字列**で中間シートに書き込んでいたのに、投稿側で読み戻すと `Sun Jul 12 2026 00:00:00 GMT+0900 (Japan Standard Time)` という壊れた表記でSlackに出た。

- 原因: Googleスプレッドシートは、`"2026/07/12"` のように日付に見える文字列をセルに入れると**自動的に日付型（Date）へ変換して保存する**。
- その後 `getValues()` で読むと、文字列ではなく **Dateオブジェクト**が返り、`String(date)` すると上記の長い英語表記になる（`getValues()` はセルの「値」を型付きで返すため）。
- **整形処理自体は正しく、"中間シートへの書き込み→読み戻し"の往復でSheetsが型を変えてしまう**のがミソ。ローカル変数で完結していれば起きず、シートを中継点にしたことで顕在化した。

### 対処

書き込み前に、その列の**表示形式をテキスト（`@`）に設定**しておくと、文字列がそのまま保持され日付変換されない。

```js
// 発生日の列だけ、書き込み前にテキスト書式にする
const dateCol = OUTPUT_KEYS.indexOf("発生日") + 1;
outSheet.getRange(1, dateCol, rows.length, 1).setNumberFormat("@");
outSheet.getRange(1, 1, rows.length, OUTPUT_KEYS.length).setValues(rows);
```

- 読み戻し側でも防御的に「Dateで返ってきたら `Utilities.formatDate(v, "Asia/Tokyo", "yyyy/MM/dd")` で整形」を入れておくと、誰かが手でセルを日付に直しても壊れない。
- 教訓: **シートを中継したデータは、書いた型のまま返ってくるとは限らない**。特に日付・数値・先頭ゼロ付き文字列（電話番号など）はSheetsの自動型変換に注意。文字列として保持したいならテキスト書式が確実。

## 4. 「暗黙の位置依存」を作ってしまう設計に注意（列順とフォールバックの整合）

流用先の `formatVocForSlack(voc)` は、`voc["キー名"]` で値を取り出しつつ、値が無いと `Object.values(voc)` の**位置引数**（`value_array[0]`〜`[11]`）にフォールバックする実装だった。

- そのため中間シートの列順（＝再構成するオブジェクトのキー順）を、このフォールバックの並びと**一対一で揃えておく必要がある**。揃えておかないと、空セルがあったときにフォールバックが別項目の値を誤って拾う。
- こういう「並び順が暗黙の契約になっている」コードは、離れた2ファイルを同時に直さないと壊れる。今回はコード内コメントと `CLAUDE.md` に「順序を崩すな、両方揃えろ」と明記して、将来の自分・他人が踏まないようにした。
- 本来はキー名で厳密に引く方が安全で、位置フォールバックのような仕組みは新規では避けたい。既存仕様に合わせざるを得ない場合は、依存関係をドキュメント化しておくのが次善策。

## 5. worktreeでの作業→mainへマージ→本体でclasp push、という反映フロー

`.clasp.json`（scriptIDを含む）は `.gitignore` 対象で本体（mainチェックアウト）にしか無く、git worktree には複製されない。よってworktreeから直接 `clasp push` はできず、**worktreeでコミット → mainへマージ → 本体で `clasp push`** の順で反映する。

- 途中でまた `clasp push` が `A file with this name already exists in the current project: appsscript` で失敗した。原因・対処は以前まとめた [2026-07-14: clasp v3のコマンド変更と、git worktreeが引き起こす.claspignoreの落とし穴](2026-07-14-clasp-v3-and-claspignore.md) と同じ（worktreeが `.claude/worktrees/` 配下にあり、`skipSubdirectories:false` のためclaspが各worktreeの `appsscript.json` まで二重に拾う）。`.claspignore` で `.claude/**` を除外すると解消。
- 同じ落とし穴を再度踏んだので、今回はプロジェクトの `CLAUDE.md` 側にもこの前提を書き足して、プロジェクト単位でも参照できるようにした。

## 6. ドキュメント（README / CLAUDE.md）の同時更新も"作業のうち"

コードを足したら、`README.md`（何をする機能か・手順）と `CLAUDE.md`（触ると壊れる非自明な前提）も同じPR/コミット群で更新する。

- 特に今回のような「発生日はテキスト書式で保持」「列順はフォールバックと揃える」といった**非自明で再発しやすい前提**は、コメントだけでなく `CLAUDE.md` に集約しておくと、次にAIや他人が触るときの事故を防げる。
- README（利用者向け説明）と CLAUDE.md（改修者向けの地雷マップ）は読者が違うので、両方に別の粒度で書くのが良い。

## 7. まとめ

- GASの機能追加は、別プロジェクト複製より「同一プロジェクトにファイル追加＋既存関数の流用」が基本。既存関数の拡張は後方互換デフォルトで。
- **Sheetsを中継点にすると型が変わる**。日付に見える文字列は勝手にDate化するので、文字列で保持したい列はテキスト書式（`@`）にする。読み戻し側でも型を正規化しておくと堅い。
- 位置依存・順序依存の暗黙契約は事故のもと。避けられないなら必ずドキュメント化。
- worktree×claspの落とし穴は「学習済みでも再発する」種類のものなので、プロジェクト直下の `CLAUDE.md` にも前提を残すと効く。
